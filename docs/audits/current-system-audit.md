# 晨亿达 ERP 当前系统技术审计

审计日期：2026-07-11

审计范围：根仓库、本地 Python/SQLite ERP、物料治理落地包、在线 Sites/Vinext/D1 项目、现有测试与已保存生产版本
审计方式：源码与文档阅读、Git 拓扑核验、本地 SQLite 只读检查、Sites 元数据只读检查；未修改数据库、未调用生产业务写接口、未部署

## 1. 结论摘要

当前仓库已经存在两套可运行 ERP：

1. `chenyida_erp_app/` 是功能覆盖较广的本地单体应用。它以 Python `ThreadingHTTPServer`、SQLite 和静态网页实现，核心业务闭环确实存在，并有一套较完整的冒烟测试。
2. `chenyida_erp_site/` 是当前公开网址对应的多人在线实现。生产版本 3 来自提交 `2b4f1787ddbc7e0941ab2d5f5cadea6e817e8f12`，使用 Vinext、Cloudflare Worker 和 D1。它迁移了大部分页面和业务流程，但数据模型、物料匹配、审批、迁移和测试深度明显低于设计规格。

物料主数据中心 V2 不能建立在现有通用 JSON 记录继续扩张的基础上。应先确认在线 Site 为唯一生产主系统，修复源码可恢复性，建立版本化迁移和关系化物料模型，再迁移匹配、审核、替代料和客户专用控制。

当前最严重的风险是：

- 本地数据库 9 个预置账号全部仍匹配代码和文档中的默认密码。
- 根仓库把在线 Site 记录为 gitlink，但没有 `.gitmodules`，新克隆无法恢复在线源码。
- 本地 SQLite 没有外键、迁移版本或生产级并发策略；在线 D1 又把多数业务对象放在 `erp_records.data_json` 中，缺乏关系约束。
- 在线物料导入没有实现设计中的自动、疑似和冲突匹配，当前每一行都按“新物料”进入待处理。
- 在线备份保存在同一个 D1 的 JSON 记录中，且恢复限制为不超过 80 条记录与余额，不能作为生产灾备。

### 2.2 源码拓扑问题

根仓库对 `chenyida_erp_site` 的记录模式为 `160000`，即 gitlink，目标提交为 `9f2c2dca9ccde237cb2db6c01d2e3792b284e6e9`。但是根目录没有 `.gitmodules`，嵌套 Site 仓库也没有配置 Git remote。

因此需要区分两种结论：

- **当前本机是否有完整在线 Site 源码：有。** `chenyida_erp_site/` 中存在前端、API、D1 schema、迁移、测试和生产提交历史。
- **根仓库是否能从新克隆完整恢复在线 Site 源码：不能。** GitHub 根仓库只保存了不可解析的 gitlink，没有子模块地址，也没有内嵌 Site 文件内容。

这属于高优先级可恢复性问题。V2 开发前必须选择以下一种方式并记录：

- 将 Site 源码直接纳入根仓库；或
- 建立真实、可访问的独立 Site 仓库并补齐 `.gitmodules`。

## 5. 当前数据库结构

### 5.1 本地 SQLite

数据库文件：`chenyida_erp_app/data/erp.sqlite3`，审计时大小约 228 KiB。只读检查结果：

- `PRAGMA integrity_check`：`ok`
- SQLite：3.28.0（实际应用运行应使用项目指定的 Python 3 运行时）
- journal mode：`delete`
- `foreign_keys`：0
- `user_version`：0
- schema version：26

当前有 26 张表：

- 身份与审计：`app_users`、`app_sessions`、`activity_log`
- 主数据：`items`、`supplier_mappings`、`cleaning_rows`、`products`、`customers`、`suppliers`
- BOM：`product_boms`、`bom_lines`
- 采购库存：`purchase_orders`、`purchase_order_lines`、`inventory_balances`、`inventory_transactions`、`inventory_adjustments`
- 生产：`work_orders`、`work_order_materials`、`production_reports`
- 销售：`quotations`、`sales_orders`、`shipments`
- 品质：`quality_inspections`、`quality_defects`
- 财务：`financial_documents`、`financial_payments`

审计时主要记录数：

| 对象 | 记录数 |
| --- | ---: |
| 用户 | 9 |
| 会话 | 1 |
| 活动日志 | 9 |
| 内部物料 | 4 |
| 供应商映射 | 2 |
| 待清洗记录 | 0 |
| 客户 | 1 |
| 供应商 | 2 |
| 产品 | 1 |
| BOM / BOM 明细 | 1 / 3 |
| 库存余额 | 4 |
| 采购、生产、销售、财务、品质业务记录 | 0 |

本地 schema 的主要问题：

- 所有关联字段都没有外键定义。
- 只有 `CREATE TABLE IF NOT EXISTS`，没有版本化迁移或升级路径。
- `PRAGMA user_version=0`，无法判断数据库处于哪个应用版本。
- 多个数量、单位、价格和交期字段在物料映射中仍用文本表示。
- `activity_log` 没有操作者、结果和请求编号，审计追踪不足。
- `app_sessions` 保存可直接使用的原始会话令牌。

### 5.2 在线 D1

源码和迁移声明 8 张表：

- `app_meta`
- `app_users`
- `app_sessions`
- `audit_log`
- `erp_records`
- `inventory_balances`
- `inventory_transactions`
- `idempotency_keys`

其中绝大多数业务对象并没有独立关系表，而是写入：

```text
erp_records(id, kind, code, data_json, version, created_by, created_at, updated_at)
```

`kind` 当前覆盖物料、映射、清洗、产品、客户、供应商、BOM、采购、盘点、生产、报价、销售、品质、财务和备份等 22 类记录。

在线模型的优点是迁移快、带通用版本号，并用独立余额/流水支持关键库存操作。缺点是：

- D1 无法用外键、列类型、检查约束和索引保护大部分业务字段。
- 关联关系、状态值、金额和数量主要依赖 TypeScript 运行时。
- 查询、报表、去重、迁移和数据质量检查需要解析 JSON。
- `db/schema.ts`、Drizzle 迁移和 `SCHEMA_STATEMENTS` 同时定义结构，存在多源漂移风险。
- 运行时 `ensureSchema()` 仍会启动时建表，不是严格的版本化迁移。
- 迁移文件与运行时声明已有细微差异，例如库存非负 `CHECK` 只在运行时声明中出现，已有生产表不会自动获得该约束。

本次审计未使用管理员账号查询生产业务数据，因此这里只确认了生产版本来源和源码声明结构，没有把生产 D1 的实际记录数当作已核实事实。

## 6. 当前物料治理能力

### 6.1 已具备

- 内部物料编码与基本字段
- 供应商原始名称、料号到内部物料的映射
- 供应商 CSV 批量导入
- 电阻、电容、连接器、PI 和部分封装/电压/规格解析
- 本地版基于精确供应商映射和加权相似度的候选评分
- 本地版自动、疑似、新物料分流
- 确认供应商映射和从清洗记录创建物料
- BOM 只接收内部物料编码
- 治理规则、字段字典、SOP、CSV/Excel 模板和示例清洗工具

### 6.2 尚不足

- 在线导入只识别粗品类和封装，没有调用已有物料库或供应商映射做候选匹配。
- 在线新物料编码为 `MAT-品类-日期-随机串`，本地和治理规范为 `CYD-品类-6位流水`，已经出现主编码分叉。
- 本地匹配算法只支持少量品类和正则，缺少单位、品牌/MPN、环保、客户限制和冲突检测。
- 没有独立的品类和属性定义表，无法强制每类物料的必填属性。
- “新物料审核”实际上是拥有 material 权限的用户一次提交直接建档，不是多岗位审批。
- 没有替代料、客户专用料、单位换算、重复合并和数据质量指标的应用实现。
- 本地模板、SQLite 和在线 D1 之间没有正式同步或迁移流程。

## 7. 安全与数据风险

| 严重度 | 风险 | 影响与证据 | 建议 |
| --- | --- | --- | --- |
| 严重 | 本地默认密码未更换 | 只读哈希比对确认 9 个预置账号全部仍匹配公开文档中的默认密码 | 立即停止共享这些账号，逐个重置，强制首次改密；移除固定默认密码种子 |
| 高 | 在线源码无法由根仓库恢复 | 裸 gitlink、无 `.gitmodules`、嵌套仓库无 remote | V2 前修复源码归属与远端可恢复性 |
| 高 | 无受控数据库迁移 | 本地 `user_version=0`；在线同时使用迁移和启动建表 | 建立单一迁移权威、版本表、试迁移和恢复门禁 |
| 高 | 数据关系缺少数据库保护 | 本地无外键；在线多数业务字段为 JSON | V2 物料核心改为关系化表和约束，保留兼容读取层 |
| 高 | 在线备份不是灾备 | 快照保存在同一 D1，恢复上限 80 条，且不包含用户、审计和会话 | 建立外部加密导出、保留策略和恢复演练 |
| 高 | 在线烟测可能污染生产 | `erp-api-smoke.mjs` 可接受任意 URL，会创建用户和业务记录，无清理 | 强制测试环境标识，拒绝生产 host，提供 teardown |
| 高 | 本地 Web 安全不足 | Cookie 无 `Secure`；无 CSRF；无登录限速；异常原文返回客户端 | 仅绑定 localhost；增加 CSRF、限速、统一错误和安全响应头 |
| 中 | 会话与审计不足 | 本地保存原始 session token，活动日志无用户和请求编号 | 保存令牌摘要，扩展审计字段并设保留策略 |
| 中 | 备份质量不受控 | 本地备份与主库同盘，现存一个零字节备份 | 校验文件大小和完整性，异地保存，定期恢复验证 |
| 中 | 数据模型和编码分叉 | 本地 `CYD-*`，在线 `MAT-*`；字段和状态也不完全一致 | 人工确认唯一编码和字段字典后再迁移 |
| 中 | 权限粒度低 | 多数角色按模块授权，没有提交、审核、执行、作废分离 | V2 引入动作级权限和审批职责分离 |
| 中 | 公开仓库隐私风险 | 根仓库包含公司资料、模板和可能的业务样例；历史公开后难彻底撤回 | 建立公开/内部仓库边界和敏感文件检查 |

## 8. V2 可能影响的模块

物料主数据是多个模块的引用根，V2 不能只修改“物料页面”。影响范围如下：

| 模块 | 影响 |

## 9. 与在线设计规格的主要偏差

- 规格要求模块化服务和数据访问层；实际在线 API 仍集中在约 1400 行的 `app/lib/erp-api.ts`。
- 规格要求关系数据库和模块边界；实际多数对象使用通用 JSON 表。
- 规格要求动作级权限；实际为模块级权限集合。
- 规格要求统一状态转换和不可原地改写；实际多个对象可直接创建为“已审核”等任意状态，缺少统一转换服务。
- 规格要求来源维度登录限速；实际只按账号统计失败次数。
- 规格要求首位管理员首次改密；实际初始化管理员设置为无需改密。
- 规格要求旧数据专用迁移工具；尚未实现。
- 规格要求单元、集成、端到端和迁移测试；当前 `npm test` 只执行构建和两项静态产物断言。
- 规格文件状态仍写“待书面规格审阅”，但系统已经发布到生产，文档门禁和实际发布脱节。
