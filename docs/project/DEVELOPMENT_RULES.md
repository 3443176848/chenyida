# 晨亿达ERP开发规则

本文件是 `AGENTS.md` 的流程化补充。若两者冲突，以更严格的规则为准。

## 开发前必须阅读

1. `AGENTS.md`
2. `docs/project/MASTER.md`
3. `docs/project/TASKS.md`
4. `docs/project/PROJECT_CONTEXT.md`
5. `docs/project/DECISIONS.md`
6. 当前任务文档

最高优先级规则：**先阅读 `MASTER.md`，再继续当前任务。**

## 标准流程

```text
读取 MASTER
  -> 确认 TASKS 中唯一当前任务
  -> 恢复 PROJECT_CONTEXT
  -> 阅读当前任务与决策
  -> 检查 Git 状态
  -> 开发
  -> 测试
  -> 更新 MASTER / TASKS / CHANGELOG / STATUS
  -> 独立 Commit
  -> 等待人工确认
```

## 强制规则

1. 每次只完成一个任务编号。
2. 禁止扩大任务范围。
3. 禁止修改无关代码或覆盖用户未提交变更。
4. 数据库修改必须提供版本化迁移、恢复方案和隔离迁移测试。
5. 所有新功能必须有与风险相称的单元、集成或端到端测试。
6. 完成任务必须更新 `MASTER.md`、`TASKS.md`、`CHANGELOG.md` 和 `STATUS.md`。
7. 每个任务必须创建独立、聚焦、可审阅的 Git Commit。
8. 未经明确确认不得修改生产环境、生产数据、生产访问策略或执行部署。
9. 不删除历史数据；已过账记录通过调整、冲销或反向记录更正。
10. 重要写操作必须记录操作者、动作、对象、结果、请求编号和时间。
11. AI 结果不得直接覆盖正式数据，物料不得未经审核自动合并。

## 任务边界

- 任务开始时把目标、非目标、依赖、验收和禁止事项写入 `TASKS.md` 或独立任务文档。
- 发现相关问题但不影响当前验收时，记录为新任务，不顺手修复。
- 阻断时记录阻断条件、证据、责任人和解除标准，不用猜测替代业务决定。

## 数据库与生产保护

- 测试和本地开发不得连接生产 D1 或写生产 URL。
- 迁移必须采用扩展、回填、切换、收缩步骤；已执行迁移不得修改。
- 生产迁移前必须有快照、只读试迁移、数量核对、恢复路径和人工授权。
- `.env`、密码、令牌、数据库、备份、日志和客户/供应商隐私不得提交。

## 测试与验收

- 先运行受影响模块测试，再运行适用基线测试。
- 失败不得通过降低断言、跳过或写死结果绕过。
- 环境导致无法验证时，必须记录命令、错误、未验证范围和风险。
- 提交前检查 `git diff --check`、敏感信息、业务代码/迁移/部署配置是否越界。

## 文档更新责任

- `MASTER.md`：更新当前状态、任务、下一任务和风险。
- `TASKS.md`：更新状态、负责人、时间、依赖和验收结果。
- `CHANGELOG.md`：记录 Commit 及功能、数据库、API、文档变化。
- `DECISIONS.md`：记录已确认或提议的重大决策。
- `PROJECT_CONTEXT.md` / `ARCHITECTURE.md`：架构或事实基线变化时更新。
- `STATUS.md`：用同一统计口径刷新代码、目录、数据库、API、页面、测试和 Git 状态。
