# 物料主数据中心 V2 分阶段实施计划

日期：2026-07-11

前提：在线 Site 是候选生产主系统；最终决定见 `business-decisions.md`
原则：新增式迁移、兼容旧数据、先试迁移后切换、未经授权不写生产

## 1. 目标与非目标

### 1.1 目标

- 建立唯一内部物料权威源。
- 用品类属性定义代替不可校验的自由文本。
- 支持供应商料号、品牌/MPN、历史名称和客户料号映射。
- 支持导入批次、标准化、候选评分、冲突拦截和逐行结果。
- 建立新物料多角色审核、生命周期和完整审计。
- 支持单位换算、替代范围和客户专用限制。
- 让 BOM、采购、库存、生产和品质通过稳定接口引用物料。
- 提供 SQLite、D1 JSON 和治理模板的可核对迁移路径。

### 1.2 非目标

- 本计划不同时重做销售、财务、生产排程或全部 ERP UI。
- 不在第一阶段引入机器学习服务；先使用可解释规则和确定性评分。
- 不一次性清洗全部历史低频物料。
- 不在迁移完成前删除旧表、旧 JSON 记录或旧编码。
- 不把生产数据复制到公开仓库、普通测试环境或日志。

## 2. 目标架构

### 2.1 服务边界

V2 建议拆分为以下服务边界，仍可部署在同一 Worker 中：

1. **物料目录服务**：内部物料、品类、属性、状态和版本。
2. **标识与映射服务**：供应商料号、品牌/MPN、别名和客户料号。
3. **导入服务**：批次、原始行、解析结果、逐行错误和重试。
4. **匹配服务**：标准化键、候选、评分、冲突和决策证据。
5. **审核服务**：申请、节点、角色、结论、驳回、补资料和启用。
6. **适用性服务**：单位换算、替代料、客户和产品范围限制。
7. **迁移与质量服务**：旧数据映射、核对、重复检测和质量指标。

浏览器通过 API 使用这些能力。业务规则、权限、状态转换和编码生成全部在服务端执行。

### 2.2 建议数据对象

V2 使用关系化核心，不继续把以下对象只保存在 `erp_records.data_json`：

| 对象 | 关键职责 |
| --- | --- |
| `material_categories` | 品类编码、名称、状态和编码前缀 |
| `material_attribute_definitions` | 品类属性、类型、单位、必填、规范化规则和排序 |
| `materials` | 内部编码、标准名称、品类、基础单位、生命周期、版本和审计字段 |
| `material_attribute_values` | 每个物料的结构化属性值和规范化值 |
| `material_aliases` | 历史名称、旧编码和合并后的别名 |
| `supplier_material_mappings` | 供应商料号、品牌/MPN、采购单位、换算、MOQ、交期和状态 |
| `material_units` | 基础单位、业务单位、换算分子/分母、精度和生效时间 |
| `material_import_batches` | 来源、文件摘要、供应商、状态、统计和操作者 |
| `material_import_rows` | 原始字段、解析结果、标准化键、处理状态和错误 |
| `material_match_candidates` | 候选物料、评分、规则证据、冲突原因和排名 |
| `material_change_requests` | 新建、修改、冻结、启用、合并申请 |
| `material_approval_steps` | 审核角色、顺序、结论、意见和时间 |
| `material_alternates` | 主料、替代料、替代类型、适用范围、审批和状态 |
| `customer_material_restrictions` | 客户料号、客户/产品范围和限制规则 |
| `material_status_history` | 状态、版本和关键变更历史 |
| `legacy_material_crosswalk` | 来源系统、旧标识、新内部编码和迁移决策 |

关键字段使用明确类型、唯一约束、外键或等价引用校验及必要索引。价格历史是否属于主数据中心由业务决策确定，不默认塞入当前映射记录。

### 2.3 生命周期建议

```text
草稿 -> 审核中 -> 已启用 -> 冻结 -> 已停用
                  |          |
                  +-> 变更审核 <-+
```

- 只有“已启用”物料可新增到 BOM 和新业务单据。
- “冻结”禁止新引用，但保留历史查询和已有单据引用。
- “已停用”不得恢复为普通可用状态，确需恢复必须发起受控变更。
- 重复物料不物理删除，通过合并和别名保留历史追溯。

## 3. 阶段 0：决策、源码与基线

### 目标

消除 V2 开发前的责任、源码和测试环境不确定性。

### 工作项

- 完成 `business-decisions.md` 中的阻断决策。
- 明确在线 Site 是否为唯一生产主系统，本地版何时转为只读参照。
- 修复根仓库裸 gitlink，确保新克隆能恢复生产源码。
- 记录生产版本、D1 迁移版本和本地 SQLite 快照摘要。
- 建立与生产完全隔离的测试 D1 和测试账号。
- 为 `erp-api-smoke.mjs` 增加生产 host 拒绝和数据清理设计。
- 建立迁移编号、执行记录、校验报告和恢复责任人规范。

### 交付物

- 已确认的业务决策记录
- 源码归属 ADR
- 测试环境说明
- 当前数据盘点报告模板
- 迁移和恢复运行手册草案

### 阶段门禁

- 新机器可完整克隆并构建 Site。
- 测试脚本无法连接生产 URL。
- 不再存在固定默认账号用于 V2 测试。

## 4. 阶段 1：关系化数据底座

### 目标

在不切换现有业务读取的情况下，建立 V2 表、迁移和数据契约。

### 工作项

- 新增关系化 V2 表、唯一约束、引用约束和索引。
- 建立迁移版本表，停止把运行时 `CREATE IF NOT EXISTS` 当作迁移。
- 实现物料编码服务，但只有最终审核通过时才分配正式编码。
- 实现品类和属性定义服务，支持文本、数值、枚举、布尔和单位化数值。
- 实现材料标准化键，明确大小写、空格、单位、数值精度和同义词规则。
- 实现 V2 只读 API 和管理 API，暂不替换现有 UI。
- 为所有写入加入版本号、操作者、请求编号和审计。

### 测试

- 空库迁移、已有 V1 数据库迁移、重复执行和中途失败恢复
- 内部编码并发唯一性
- 品类必填属性和数据类型
- 外键/引用、唯一索引和非法状态
- 乐观并发冲突

### 阶段门禁

- 迁移在隔离 D1 至少重复执行两次且结果一致。
- V1 读取路径没有变化。
- schema、迁移和运行时查询一致。

## 5. 阶段 2：导入、标准化与匹配

### 目标

替换在线端“全部新物料”的导入行为，建立可解释的候选匹配。

### 处理顺序

1. 供应商与供应商料号精确映射。
2. 品牌与 MPN 规范化精确匹配。
3. 品类、关键属性和单位归一后的确定性键匹配。
4. 标准名称与辅助属性的加权相似度候选。
5. 客户专用、冻结、冲突和单位不兼容规则拦截。

### 输出分级

- 自动建议：证据充分，可由采购批量确认；默认不直接写正式映射。
- 疑似匹配：返回最多若干候选和逐项差异，由采购/工程确认。
- 冲突匹配：禁止进入 BOM、采购收货或自动建档。
- 新物料：进入审核申请，不直接创建已启用物料。

### 工作项

- 导入批次和行级状态机
- CSV 编码、表头、类型和必填校验
- 解析插件和规范化字典
- 候选评分、规则证据和阈值配置
- 行级成功/失败/警告报告
- 幂等导入和重复文件摘要检测
- 人工确认后的规则回馈统计，不自动改变生产阈值

### 测试

- 精确映射、MPN、属性匹配和同义词
- 单位换算、缺失字段和数值精度
- 冲突供应商料号、重复 MPN、客户专用误匹配
- 大批次分页、重试、重复上传和部分失败
- 匹配结果可解释性和确定性

### 阶段门禁

- 基准样本中每个结果均可复现并说明命中规则。
- 冲突不会自动写正式物料或映射。
- 导入失败不会留下半批数据。

## 6. 阶段 3：新物料审核与生命周期

### 目标

把“一次提交直接建档”升级为职责分离的受控审核。

### 建议流程

1. 采购或工程提交申请和来源证据。
2. 系统执行查重和候选匹配。
3. 工程确认规格、用途、属性和 BOM 适用性。
4. 品质确认环保、供应商准入和检验规则。
5. 仓库确认基础单位、采购单位、包装和批次策略。
6. 财务/成本按确认范围检查计价单位和分类。
7. 数据管理员复核重复项并启用，系统生成正式编码。

### 工作项

- 可配置但受控的审核节点和角色
- 驳回、补资料、撤回、超时和转交
- 申请版本和并发控制
- 启用、冻结、停用、合并和历史追溯
- 动作级权限和完整审计
- 待办列表、审核时长和逾期指标

### 测试

- 每个角色允许和禁止的动作
- 跳步、重复审核、并发审核和账号停用
- 驳回后补资料、重新提交和版本保留
- 编码只在最终通过时生成且不复用

### 阶段门禁

- 页面和直接 API 都无法绕过审核。
- 所有状态变化有操作者、请求编号和前后值。

## 7. 阶段 4：单位、替代料和客户专用控制

### 目标

让物料不只是“可查记录”，而是能安全参与采购、BOM、库存和生产的受控主数据。

### 工作项

- 基础单位和业务单位换算，使用整数分子/分母或明确精度
- 替代类型、有效期、客户/产品/BOM 范围和审批
- 客户料号和客户专用范围
- 冻结、停用和不合格供应商的下游拦截
- BOM 添加、采购下单、收货、领料的适用性检查 API
- 违规使用的稳定错误代码和审计

### 测试

- 可换算与不可换算单位
- 完全替代、条件替代和禁止替代
- 客户专用料跨客户误用
- 冻结物料在不同业务阶段的行为

### 阶段门禁

- 新建 BOM、采购和领料无法绕过适用性校验。
- 历史单据仍可读取，不因状态变化断链。

## 8. 阶段 5：旧数据试迁移与下游切换

### 目标

把 SQLite、在线 D1 JSON 和治理模板映射到 V2，并分模块切换引用。

### 数据来源

- 本地 SQLite 26 张表中的物料及引用
- 在线 `erp_records` 中 items、mappings、cleaning 和下游引用
- `物料主数据治理落地包/templates/` 的标准物料、映射、属性、替代和客户专用数据

### 迁移步骤

1. 只读导出并记录来源摘要、数量和时间。
2. 规范化字段和单位，生成重复组、冲突组和缺失字段报告。
3. 人工确认编码交叉表和主记录。
4. 在隔离 D1 试导入，生成逐表数量、孤儿引用和汇总差异。
5. 先让 V2 与旧模型并行读取或双写，记录差异。
6. 按 BOM、采购、库存、生产顺序切换引用。
7. 稳定期结束后再提出旧写路径停用方案。

### 核对指标

- 来源物料数、V2 主物料数、重复合并数、未决冲突数
- 供应商映射覆盖率
- BOM 行映射率和孤儿物料数
- 库存物料数与数量汇总差异
- 在途采购、未完工工单和未完成销售单引用完整性

### 阶段门禁

- 未决冲突为零或有书面例外。
- 库存和关键业务引用核对通过。
- 恢复演练成功并由业务负责人签字确认。

## 9. 阶段 6：运营、性能与生产发布准备

### 目标

完成质量指标、生产防护和发布前验收，不在本计划中自动部署。

### 工作项

- 标准化率、映射覆盖率、自动建议率、冲突数、审核周期和重复物料指标
- 大批次导入性能、索引和分页
- 登录、CSRF、速率限制、会话撤销和审计保留
- 外部加密备份和恢复演练
- 角色矩阵、操作手册和现场培训
- 生产迁移窗口、冻结规则、回退条件和责任人

### 最终验收

- 自动测试、迁移测试和关键流程端到端测试通过。
- 业务负责人完成样本核对和角色验收。
- 生产数据快照、恢复路径和回退时间已验证。
- 单独取得生产迁移和部署授权后，才进入发布任务。

## 10. 兼容与切换策略

- V2 初期只新增表和 API，不删除 `erp_records`。
- 使用 `legacy_material_crosswalk` 把本地 `CYD-*`、在线 `MAT-*` 和其他旧标识映射到唯一 V2 物料。
- 下游模块先使用兼容查询返回旧接口形状，再逐步改为 V2 标识。
- 迁移期间禁止自动合并重复物料；合并必须有人工作为和审计。
- 已存在的库存和业务单据保留原标识快照，同时增加 V2 引用，防止历史展示变化。
- 只有连续核对期无关键差异后，才停止旧写路径。

## 11. 测试总矩阵

| 层级 | 必测内容 |
| --- | --- |
| 单元 | 编码、标准化、属性、单位、评分、状态机、权限、版本冲突 |
| 数据库 | 迁移、约束、索引、事务、回滚、幂等、并发 |
| 集成 | 导入到审核、审核到启用、物料到 BOM/采购/库存/生产 |
| 迁移 | SQLite、D1 JSON、模板、交叉表、重复和孤儿引用 |
| 安全 | 未登录、越权、CSRF、限速、会话撤销、敏感日志 |
| 端到端 | 多角色申请审核、冲突处理、替代料、客户专用、下游拦截 |
| 恢复 | 外部备份、恢复、回退和迁移重跑 |

## 12. 推荐的第一项开发任务

**任务名称：建立 V2 数据契约、关系化 schema 和隔离迁移测试基线。**

范围严格限定为：

- 修复 Site 源码可恢复性。
- 记录业务决策中已经确认的编码、状态和责任人。
- 新增 V2 关系表的增量迁移及 schema 定义。
- 增加迁移版本、空库/旧库/重复执行/失败恢复测试。
- 提供只读查询和示例测试数据，不接入现有 UI，不迁移生产数据。

完成标准是迁移与测试可重复，不是页面可见。
