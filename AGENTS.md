# AGENTS.md

## 适用范围

本文件适用于仓库根目录及全部子目录。子目录如需增加更严格的规则，可以新增自己的 `AGENTS.md`，但不得放宽本文件中的数据库、安全、测试和生产保护要求。

## 项目文档驱动流程

以后所有开发以仓库文档为上下文，不以聊天记录为唯一依据。任何 Codex 在开始任务前，最高优先级规则是：**先阅读 `docs/project/MASTER.md`，再继续当前任务。**

标准流程如下：

```text
MASTER.md
  -> TASKS.md
  -> PROJECT_CONTEXT.md
  -> 当前任务文档
  -> 开发
  -> 测试
  -> 更新 MASTER.md、TASKS.md、CHANGELOG.md、STATUS.md
  -> 创建独立 Git Commit
```

执行要求：

1. 开发前必须阅读 `AGENTS.md`、`docs/project/MASTER.md`、`docs/project/TASKS.md`、`docs/project/PROJECT_CONTEXT.md` 和当前任务文档。
2. 每次只执行一个任务编号；禁止扩大任务范围或修改无关代码。
3. 任务状态、依赖、责任人和验收结果必须记录在 `TASKS.md`，重大业务或架构选择必须记录在 `DECISIONS.md`。
4. 完成任务必须同步更新 `MASTER.md`、`TASKS.md`、`CHANGELOG.md` 和 `STATUS.md`，并创建独立提交。
5. 文档与代码不一致时，先核验实际代码和运行状态，再修正文档；不得为了符合文档而擅自修改生产行为。

## 项目目标

晨亿达 ERP 的目标是以内部标准编码统一物料、产品、BOM、采购、库存、生产、销售、财务和品质数据，并让多人通过受控账号协同处理业务。

当前仓库同时包含两个运行面：

- `chenyida_erp_app/`：本地 Python、SQLite 和静态网页实现，用作现场本地运行、行为参照和旧数据迁移来源。
- `chenyida_erp_site/`：在线 Sites、Vinext、TypeScript 和 D1 实现，是当前公开网址对应的生产方向。

物料主数据中心 V2 必须优先解决一物一码、属性标准化、供应商映射、批量导入、匹配分流、新物料审核、替代料和客户专用料控制。不得以新增页面代替数据规则、服务端校验和迁移设计。

## 仓库与架构规则

1. 先确认修改目标属于本地旧版、在线生产版、共享治理资料还是迁移工具，不得同时在两个运行面复制新增业务逻辑。
2. 在线生产逻辑以服务端 API 和 D1 为权威；浏览器 JavaScript 只负责交互，不得承担权限、编号、状态转换、余额或审核规则的最终判断。
3. 本地 Python 版在 V2 期间默认作为兼容和迁移来源。除非有书面决策，不得把它继续扩展成与在线版并行演进的第二套主系统。
4. 共享业务规则应放在边界明确、可独立测试的模块中。禁止继续扩大单文件 API 或在多个处理分支中复制同一规则。
5. 物料主数据、供应商映射、导入批次、匹配候选、审核、单位换算、替代关系和客户限制必须有明确数据边界，不得全部塞入无约束 JSON。
6. 所有业务引用必须使用稳定的内部标识。供应商料号、原始名称和客户料号只能作为映射或别名，不能成为 BOM、库存或生产的主键。
7. 关键写操作必须在单一事务中完成业务记录、联动记录、版本更新和审计记录。失败时不得留下部分写入。
8. 已过账的库存、生产消耗、出货和财务记录不得原地改写；更正应使用调整、冲销或反向记录。
9. 新接口必须使用稳定错误代码、中文用户提示和请求编号。服务端异常、SQL、堆栈和敏感请求内容不得返回浏览器。
10. 根仓库必须能够从新克隆中完整恢复源码。禁止提交没有 `.gitmodules` 和可用远端的裸 gitlink；在线 Site 的源码归属方案变更前必须先记录 ADR。

## 数据库迁移规则

1. 任何结构变化都必须使用版本化、可审阅的迁移文件。`CREATE TABLE IF NOT EXISTS` 只能用于开发兼容，不能代替生产迁移历史。
2. 已执行的迁移不得修改或重排；修正必须新增后续迁移。
3. `db/schema.ts`、`drizzle/` 迁移和运行时查询必须保持一致。发现差异时先修复迁移基线，不得靠启动时建表掩盖。
4. 本地 SQLite 和在线 D1 的迁移必须分别明确适用范围、前置版本、回滚或恢复方案、数据校验和执行责任人。
5. 迁移采用扩展、回填、切换、收缩四步法。破坏性删列、重命名或类型变更不得与业务切换放在同一步。
6. 新关系必须声明外键或提供等价的服务端引用校验，并为常用查询、唯一性和待办队列建立索引。
7. 所有数量、金额、单位换算和状态字段必须有明确类型、精度、合法值及约束。不得用自由文本承载可计算数据。
8. 正式迁移前必须生成可恢复快照，执行只读试迁移并核对表数、记录数、重复项、孤儿引用、库存汇总和关键金额。
9. 数据回填必须幂等、可断点续跑并生成逐行结果。禁止在部署启动时自动迁移生产业务数据。
10. 生产数据库只能通过受控流程修改。测试、烟测和本地开发不得连接或写入生产 D1。

## 测试规则

1. 业务变化至少包含对应单元测试；跨表或跨模块变化必须增加集成测试；关键用户流程必须有端到端验收清单。
2. 本地 Python 基线测试包括：
   - `server.py --self-test`
   - `smoke_test.py`
   - `go_live_check.py`
3. 在线 Site 基线测试包括：
   - `npm run lint`
   - `npm test`
   - 使用隔离测试数据库运行 `tests/erp-api-smoke.mjs`
4. 测试必须使用临时数据库、独立 D1 或明确的测试环境，并在结束后清理测试记录。任何会写数据的脚本都必须拒绝生产 URL。
5. 数据库迁移测试必须覆盖空库升级、已有数据升级、重复执行、失败回滚、约束和升级后汇总核对。
6. 物料 V2 至少覆盖编码唯一性、属性必填、精确映射、疑似匹配、冲突拦截、审核状态、单位换算、替代范围、客户专用限制和并发冲突。
7. 测试失败不得通过降低断言、跳过用例或写死结果绕过。确属环境问题时记录原因和未验证范围。
8. 文档-only 变更也要运行不写生产数据的现有测试，并确认没有修改业务代码、迁移或部署配置。

## 安全规则

1. 禁止提交真实密码、初始化凭证、会话令牌、Git Token、API Key、数据库文件、备份、日志或客户/供应商个人信息。
2. 默认账号和固定初始密码不得用于可投产环境。账号创建、首次改密、重置、停用和会话撤销必须可审计。
3. 密码只保存强哈希；会话令牌只保存摘要。生产 Cookie 必须为 `HttpOnly`、`Secure`、`SameSite=Lax` 或更严格。
4. 所有业务接口必须执行服务端认证和授权。页面隐藏按钮不构成权限控制。
5. 写接口必须考虑 CSRF、重复提交、请求大小、速率限制、输入校验、并发版本和幂等。
6. 审计日志记录操作者、动作、对象、结果、请求编号和时间，不得记录密码、令牌或完整敏感正文。
7. 备份必须与主库故障域分离，定期验证可恢复性。零字节备份、仅存于同库的快照或未经验证的导出不能视为有效灾备。
8. 公开仓库和公开 Site 不得暴露生产环境变量、数据快照、公司内部材料或无法撤销的访问凭证。
9. 任何生产访问、迁移、备份恢复、权限变化或部署都需要用户明确授权，并在执行前说明影响范围。

## Git 提交规则

1. 开始工作前检查根仓库和嵌套仓库状态；不得覆盖用户已有未提交改动。
2. 提交必须小而聚焦，使用清晰的祈使式消息，例如 `docs: audit current ERP architecture`、`feat: add material approval workflow`。
3. 业务代码、迁移、数据回填、文档和部署配置应尽量分开提交，便于审阅和回滚。
4. 数据库结构变化必须与迁移和迁移测试同一提交，禁止只提交运行时代码。
5. 不提交生成目录、依赖、运行数据库、备份、日志、截图和临时导出，除非它们是经批准的产品资料。
6. 禁止对共享分支执行强制推送、历史重写、硬重置或未经确认的文件删除。
7. 更新嵌套 Site 仓库时，必须确保源码远端可恢复，再单独更新根仓库指针并说明目标提交。
8. 提交前运行适用测试并检查 `git diff --check`、敏感信息和最终变更范围。

## 物料主数据中心 V2 完成标准

V2 阶段只有同时满足以下条件才能标记完成：

- 业务决策已确认并记录。
- 关系化数据模型和版本化迁移已通过审阅。
- 旧数据试迁移报告通过，重复和冲突有人工处置结果。
- 服务端权限、审核、幂等、并发和审计覆盖关键写操作。
- BOM、采购、库存和生产只引用有效内部物料。
- 自动测试、迁移核对和人工验收全部通过。
- 未经单独授权，不进行生产迁移或部署。
